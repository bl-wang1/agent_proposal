System

从任务边界到运行时架构，形成同一套系统协议。

ECAOS 将任务定义、交互协议、控制架构和接口契约统一起来。它不是一个单模型，而是一套把目标、世界状态、技能、验证、安全和学习准入串起来的机器人 Agent 控制平面。

Task Model

任务定义先限定系统要解决什么。

研究对象是物理世界中的长程移动操作 Agent。首阶段可以选择可控空间做验证，但系统主线不是场景分类，而是空间事实、主动感知、技能契约、验证器和安全门控如何构成同一个 agent loop。

场景可建模、可验证、可治理的物理空间；环境差异由传感器、地图、运动约束和技能实现适配。
任务巡检、找物、递送轻物、整理区域、对象转移、设备开关；每个任务包含 3-8 个子目标。
输出任务图、技能序列、检查结果、失败归因、恢复动作、完整 trace（执行证据链）和候选更新报告。
验收长程成功率、恢复率和跨域表现提升，同时不增加遗忘、安全事件、人工接管和过度拦截。
Interaction Protocol

交互是安全控制面，不是聊天附属品。

机器人执行前必须暴露目标理解、下一步动作、风险等级和确认需求；用户的确认、修改、打断和接管都写入 trace（执行证据链），成为恢复与学习的证据。

1. Draft生成任务卡和候选任务图，不执行物理动作。
2. Clarify对象、位置或权限不明确时先问清楚。
3. Armed低风险自主执行，中高风险等待确认。
4. Interrupted用户打断时冻结任务图，保持安全姿态。
5. Resolved继续、修改、回滚、接管或取消，并写回 trace。
1人机交互层目标输入、澄清确认、状态反馈、打断纠错、示教和授权。
2空间状态层地图、坐标、物体、人、状态、关系、时间和置信度。
3Agent 推理层理解目标、拆解任务、选择工具、监控执行、解释失败。
4技能执行层感知、导航、操作、沟通、恢复、回充、远程接管。
5安全治理层风险分级、权限确认、禁区、限速、力控、隐私、审计。
6评估运维层仿真、真实回放、日志、指标、失败分类、灰度发布。
Ordered Runtime Flow

信息按执行顺序进入证据链。

这里的 Contract、Trace 和 Policy 是三个系统对象：Contract 说明动作“应该满足什么”，Trace 记录执行“实际发生什么”，Policy 决定系统“接下来允许什么”。

Contract技能契约。定义一个动作的 intent、precondition、expected effect、verifier、rollback、risk level 和 failure code，例如 `place(cup, sink)` 必须满足 cup in gripper、sink reachable，并产生 inside(cup, sink)。
Trace执行证据链。记录 spatial state snapshot、空间事实、主动感知动作、skill call、gate decision、verifier result、失败边界和 human label，用于复盘和更新空间事实状态。
Policy决策与准入规则。根据 Contract 和 Trace 决定 allow、ask、inspect、block、rollback，以及候选空间事实、感知器或 active policy 能否进入 shadow/灰度。

02Spatial Snapshotfacts, confidence低置信度先 inspect
03Task Graphprecondition, effect计划只是候选
04Gate + Skillallow / ask / block动作前独立门控
05Verifypredicate result不相信裸 success

Contractprecondition / effect / rollback / adapter test

执行顺序信息从 Task Card 进入 Spatial Snapshot、Task Graph、Gate + Skill、Verify，最后沉淀为 Trace + Learn。
证据类型Contract 约束动作，Trace 记录结果，Policy 控制风险与发布。
替换模块不改协议planner、技能和 verifier 可以替换，但 evidence chain 和准入逻辑保持稳定。
Spatial Intelligence

空间能力不是一次性建图，而是可主动补充、可验证、可更新的事实状态。

空间事实状态 用拓扑图 + 语义对象图 + 局部几何摘要表示空间：房间、区域、通路、物体、遮挡、可达性、可操作面和时间戳。
主动空间感知 当遮挡、错检、关系冲突或可达性未知时，主动换视角、inspect、低风险探测或请求标注。
空间更新演进 每次任务产生 trace、spatial failure label 和 world diff；候选空间事实或感知更新先回放验证，再进入 shadow 和灰度。
空间事实格式 每条 SpatialFact 保留 subject、predicate、object、frame、confidence、evidence、expires_at 和 risk_relevance；过期或低置信事实必须重新观察。
Observe Pipeline

Observe 把原始观测变成可决策的空间事实。

这一层不只是建图或写日志，而是把传感器输入、技能反馈和人工标注规范化为带证据、置信度、过期时间和风险语义的 SpatialFact。

步骤输入/动作输出为什么重要
1. Raw observation RGB-D/video、机器人位姿、地图、技能反馈、历史 memory 和 human label。 可追溯的 observation packet。 所有后续空间事实都能回到具体证据源。
2. Spatial candidates 抽取对象、区域、局部几何、遮挡、可达性和 affordance 候选。 visible、reachable、on/inside/near、occluded_by、placeable_on 候选。 把“看见什么”转成 planner 和 verifier 能使用的空间谓词候选。
3. SpatialFact write 为每个候选写入 subject、predicate、object、frame、confidence、evidence、source_viewpoint 和 expires_at。 带证据和生命周期的 SpatialFact。 空间状态不再是静态 3D memory，而是可查询、可过期、可复核的 Agent state。
4. Conflict / expiry / risk 新旧事实冲突时写入 conflict_set；过期或低置信事实降权；高风险相关事实进入 gate。 inspect、change_viewpoint、probe_reachability、ask_human 或 defer。 错误记忆不会被直接覆盖或直接驱动高风险动作。
5. Decide / Verify handoff Planner 读取 SpatialFact Graph；verifier 用独立证据检查 expected effect。 下一步计划、主动感知动作或 hidden spatial failure 标记。 Observe 的产物直接影响执行选择和空间验证，而不是只作为背景上下文。
Core Contracts

系统靠契约连接，而不是靠 prompt 默契。

Skill Contractintent、precondition、effect、verifier、rollback、risk level、timeout、failure code。
SpatialFact Graphvisible、reachable、on/inside/near、occluded_by、path_clear 等空间事实，以及 confidence、evidence 和 expiry。
Active Perception Policy根据 uncertainty、expected information gain、动作成本和风险，选择 inspect、change_viewpoint、probe、ask_human 或 defer。
Trace Schematask graph、memory snapshot、skill I/O、predicate result、failure boundary、promotion decision。
Runtime Interfaces

模块边界必须能写成稳定 API。

第一版实现不追求复杂微服务，而是先把各模块的输入输出固定下来；后续替换 planner、VLA、导航栈或 verifier 时，不改变 trace 和治理协议。

接口输入输出责任边界
Planner.plan(ctx) 用户目标、约束、世界快照、历史失败、可用技能。 带 precondition、effect、verifier、rollback 的 task graph。 只生成候选计划，不拥有安全放行权。
SpatialState.snapshot() 地图、传感器观测、记忆事实、用户偏好、时间戳。 带 SpatialFact Graph、confidence、evidence 和 expires_at 的 SpatialSnapshot。 只提供可审计状态假设，低置信度事实不能直接驱动高风险动作。
SkillRegistry.select(intent, world) 任务意图、对象状态、机器人本体、技能统计。 SkillContract 和具体 implementation 版本。 迁移共享 intent/contract；本体相关 controller 作为 implementation 替换。
Governance.check(step) 技能契约、风险等级、路径、权限、禁区、用户状态。 allow、ask、inspect、block、rollback 决策和证据。 独立于 planner；硬约束、急停和近人保护不由语言模型裁决。
Verifier.check(effect, observation) 预期效果、传感器观测、空间事实候选更新。 predicate result、confidence、failure boundary。 判定任务是否推进，不等同于技能是否返回 success。
TraceStore.write(trace) episode 中所有计划、动作、门控、验证、恢复和接管事件。 可回放 trace、失败标签、候选更新证据。 作为实验指标、持续学习和发布审计的唯一事实来源。
Minimum Stack

首版实现用可替换组件，不绑定单一模型路线。

实现策略是先把 runtime、trace、verifier 和治理协议跑通；低层导航、操作和学习策略用现成机器人栈包成 SkillContract，避免第一版同时发明系统层和控制层。

Agent runtime Python orchestration + typed schema；优先实现任务图、门控、trace、回放和评估，不先追求复杂训练管线。
Robot skills 导航/操作技能可由 ROS、经典控制器、OpenVLA/Octo/LeRobot policy 或脚本 fallback 包装为 SkillContract。
World model 首版从 2D/3D semantic memory、对象表和时间戳事实开始；低置信度事实触发 inspect，不直接执行。
Eval ops 所有 episode 写入统一 trace；指标、失败标签、shadow evaluation 和 candidate admission 共享同一日志格式。
