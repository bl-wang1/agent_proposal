Spatial Agent
总览 系统框架 运行闭环 实验验证 路线与风险 参考
Task Definition

先把“机器人 Agent 要做什么”定义清楚

研究对象是物理世界中的长程机器人 Agent 任务，而不是单步抓取或单一导航 demo。任务必须同时考验空间感知、空间事实状态、规划、技能执行、进度验证和安全治理。

场景机器人需要在真实物理空间中理解对象、区域、路径、遮挡、可达性和风险；具体场景只作为实验载体。
任务巡检、找物、递送轻物、整理指定区域、对象转移、设备开关。每个任务 3-8 个子目标。
输入语音/文本目标、手机/平板任务卡确认、环境观测、空间记忆、安全约束和用户偏好。
输出任务图、技能序列、检查结果、失败归因、恢复动作、完整 trace 和候选更新报告。
Generalization Boundary

室内/室外不是系统边界，治理接口才是系统边界

Spatial Agent 的研究对象不是某一个场景，而是“目标 - 世界状态 - 技能执行 - 验证 - 学习准入”的闭环协议。不同环境替换底层感知、地图、运动和技能实现，但不改变任务图、契约、验证和安全治理逻辑。

跨环境不变量 任务图、子目标谓词、技能契约、风险分级、进度验证、失败归因、trace schema、候选发布门槛。
环境适配项 传感器组合、地图表达、定位方式、运动约束、可用技能库、通信链路、天气/光照/地面条件。
首阶段验证域 首阶段选择可控任务是为了减少不可控变量，优先验证空间事实图、主动感知、验证器和技能契约是否成立。
跨域扩展路径 在园区配送、仓储搬运、户外低速巡检等域中复用上层协议，替换导航、感知、安全半径和技能实现。
Transfer Axes

泛化要拆成三个可测迁移轴

“跨环境、跨物体、跨本体”不能混在一个平均成功率里。Spatial Agent 需要分别证明上层任务图和技能契约能复用，底层 adapter 能替换，并且失败能被归因到正确边界。

迁移轴保持不变允许变化主要失败边界
跨环境 任务目标、成功谓词、技能契约、风险等级和发布门槛。 地图、光照、障碍物、人流、通信质量、可达路径和定位方式。 world_state_error、navigation_failure、stale_memory。
跨物体 操作意图、容器/支撑/开关等关系谓词、验证逻辑。 外观、大小、材质、抓取姿态、遮挡和相似干扰物。 perception_error、affordance_error、verifier_uncertain。
跨本体 task graph、SkillContract、failure code、trace schema 和 verifier 目标。 机械臂自由度、末端执行器、导航底盘、控制器、动作空间和安全半径。 implementation_mismatch、controller_failure、safety_margin_error。
Task Suite

任务集必须能复现长程失败，而不是只展示成功样例

首轮实验使用 10-20 个长程任务，按任务机制而不是场景名字划分；每类任务都要给出初始状态、子目标谓词、成功条件、失败标签和恢复选项。

任务族示例任务主要考察能力测试划分
巡检与状态确认 检查会议室门窗、电源、白板和桌面遗留物。 空间覆盖、状态谓词、低风险自主导航、异常上报。 train: 常见布局；test: 物体位置扰动；holdout: 未见房间。
找物与定位 找到用户指定杯子、工具、文件夹或包裹，并返回位置证据。 目标澄清、视觉定位、记忆时效、低置信度 inspect。 train: 已知物体；test: 未见摆放；holdout: 相似干扰物。
递送与对象转移 把轻物从 A 点送到 B 点，必要时等待用户确认接收。 导航、抓取、路径风险、交互确认、交接状态验证。 train: 固定路线；test: 动态障碍；holdout: 新收件人/新区域。
整理与清洁前置 把桌面杯子放入水槽，再擦拭指定桌面区域。 任务图、容器关系验证、失败恢复、后续步骤依赖。 train: 单对象；test: 多对象；holdout: 容器边缘/遮挡失败。
设备与权限动作 开关灯、按按钮、操作低风险设备或进入受限区域前确认。 权限、不可逆动作识别、安全门控、用户接管。 train: 明确授权；test: 模糊指令；holdout: 权限冲突。
Failure Labels

失败必须被标注到可行动的系统边界

world_state_error 地图、物体位置、关系或 affordance 过期；恢复策略是 inspect、重新定位或降低记忆置信度。
planning_error 任务图顺序错误、缺少前置条件或恢复循环；恢复策略是重新分解目标并加入 verifier 约束。
skill_failure 抓取、放置、导航或设备操作失败；恢复策略是换角度、换技能版本、rollback 或请求协助。
governance_event 权限、禁区、近人、过度拦截或人工接管触发；恢复策略是解释原因、调整风险阈值或终止任务。
MVP Acceptance

最小可验收版本要满足五件事

1用户能自然给目标，系统能复述任务和对象。
2执行前展示下一步动作、风险等级和是否需要确认。
3观察和导航可自主；抓取、敏感区域、设备操作需要确认。
4不确定时停下来询问，而不是靠猜继续执行。
5失败后能解释原因，并给出重试、换角度、请求协助或安全暂停。
Example Episode

杯子入水槽后擦桌面

这个任务包含对象转移、容器关系验证和后续清洁，能暴露进度误判、技能选择和恢复策略问题。

  1. 生成任务图：grasp(cup) → place(cup, sink) → verify → wipe(table)。
  2. 执行前确认对象和风险；靠近人身或不可逆动作需要确认。
  3. place 后 verifier 检查 object_inside(cup, sink)。
  4. 若杯子停在边缘，诊断 near-edge placement，调用 regrasp + place_deeper。
  5. trace 写入失败账本，用于训练 recovery policy 和更新 affordance memory。
