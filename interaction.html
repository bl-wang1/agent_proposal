Interaction Protocol

先定义人机协作协议，再扩大机器人自主性

物理世界 Agent 的交互不是聊天界面附属品，而是安全控制面的一部分。系统必须让用户看见目标、下一步动作、风险等级和确认需求，并支持随时打断、纠错和接管。

Before Action

执行前，机器人必须把自己要做什么说清楚

交互层的核心职责是让意图、对象、风险和确认状态可见，避免机器人“默默理解错然后执行”。

目标复述把用户目标转换成任务卡：对象、位置、预期结果、完成条件。
下一步动作展示即将执行的 skill、目标对象、预计路径或操作区域。
风险等级标记低/中/高风险，并说明风险来源：靠近人、敏感区域、不可逆动作或设备操作。
确认状态明确哪些步骤可自主，哪些步骤必须用户确认或远程接管。
Task Card

用户确认的对象不是一句话，而是一张可审计任务卡

任务卡把自然语言目标转成系统状态：用户能看到系统理解、下一步动作、风险来源和可选操作；每次确认、拒绝、修改或打断都会进入 trace。

字段用户看到系统记录用途
goal_summary “把桌上的杯子放进水槽，然后擦桌面。” 目标、对象、位置、完成条件和用户约束。 防止目标理解错误，并支持后续复盘。
next_action “准备抓取 cup_03，目标位置 sink_01。” skill name、参数、路径、预计效果和 verifier。 让用户确认的是具体动作，而不是模糊意图。
risk_notice “中风险：靠近水槽边缘，可能需要重新放置。” 风险等级、触发规则、所需权限和安全门控证据。 把安全决策转成用户可理解的解释。
available_choices 允许、换目标、先观察、远程接管、取消。 用户选择、时间戳、原因、是否改变任务图。 把人类反馈转成训练 ask-human/recovery policy 的事件。
Interaction State Machine

交互状态必须可恢复、可回放、可统计

1. Draft系统生成任务卡和候选任务图，不执行物理动作。
2. Clarify对象、位置或权限不明确时，先问清楚再进入计划。
3. Armed低风险步骤自动执行；中高风险步骤等待确认。
4. Interrupted用户打断时冻结任务图，保持安全姿态并记录原因。
5. Resolved用户选择继续、修改、回滚、接管或取消，状态写回 trace。

Ask-human policy

这个策略独立于 planner。planner 可以提出动作，但是否需要问人由交互/安全层共同决定。

def should_ask_human(step, world, risk, confidence):
    if user_interrupted():
        return Ask(reason="user interruption")

    if step.is_irreversible:
        return Ask(reason="irreversible action")

    if risk.level == "high":
        return Ask(reason=risk.reason)

    if world.near_human(step.path, distance="1.0m"):
        return Ask(reason="near human")

    if step.touches_sensitive_area or step.operates_device:
        return Ask(reason="permission required")

    if confidence.object_identity < 0.85:
        return Ask(reason="ambiguous object")

    if confidence.expected_effect < 0.70:
        return Ask(reason="uncertain outcome")

    return AutoProceed()
不确定就问对象不明确、目标区域不明确、结果不可逆、靠近人身或进入敏感区域时，系统不能靠猜。
问人不是失败ask-human 是长程任务成功策略的一部分，应作为指标记录，而不是被简单视为系统能力不足。
安全独立于大模型急停、避障、碰撞保护、禁区和限速必须由外部控制层执行，不能只靠 prompt。
Interrupt & Repair

用户打断后，系统要进入可恢复状态

停立即停止当前动作，保持安全姿态，记录 interruption event。
不是这个冻结当前选择，重新询问对象，更新 object identity confidence。
先别动暂停任务图，保留空间事实状态和当前假设，等待用户继续或修改目标。
放回去调用 rollback skill，把对象恢复到最近安全位置。
以后放这里写入偏好或场景规则，但标记 source=user_instruction 和 stale_after。
Failure Productization

失败后不要只报错，要给用户可选路径

失败是持续学习信号，也是用户体验的一部分。系统需要解释原因、展示风险，并提供有限、明确的下一步选项。

  1. 说明失败边界：例如“杯子没有完全进入水槽，停在边缘”。
  2. 说明可能原因：遮挡、抓取姿态、目标容器边缘、传感器置信度低。
  3. 给出恢复选项：重试、换角度、请求协助、安全暂停、取消任务。
  4. 记录用户选择和恢复结果，用于更新 recovery policy。
